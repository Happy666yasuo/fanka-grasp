# 2026-04-27 日志：CausalExplore system-level evaluation

## 今日完成内容

今天在 `embodied_agent` 中完成了 CausalExplore system-level evaluation 的第一版闭环评估能力，核心目标是把原先“planner 是否插入 probe”的评估，推进到“系统 demo 在成功场景、失败恢复场景下如何表现”的可记录、可复现对比。

主要改动如下：

1. 新增 system eval runner
   - 新增 `src/embodied_agent/causal_explore_system_eval.py`。
   - 复用 `run_mock_system_demo(...)`，对每个 strategy 的每个 episode catalog 运行固定 system scenarios：
     - `blue_success`
     - `red_place_failure`
   - 输出 run-level `system_eval_summary.json`。
   - 输出 per-strategy `system_metrics.jsonl`。
   - 统计字段包括：
     - `system_eval_case_count`
     - `success_count`
     - `success_rate`
     - `failure_count`
     - `replan_count`
     - `probe_step_count`
     - `replanned_probe_count`

2. 接入现有 CausalExplore eval
   - `run_causal_explore_eval(...)` 在生成 planner eval 后继续生成 system eval。
   - `evaluation_summary.json` 新增顶层 `system_eval_summary_path`。
   - 每个 strategy summary 新增：
     - `system_eval_case_count`
     - `system_success_rate`
     - `system_failure_count`
     - `system_replan_count`
     - `system_probe_step_count`
     - `system_metrics_path`
   - 保持原有 artifact-level、planner-level 字段兼容。

3. 更新 Markdown report
   - `evaluation_report.md` 新增 `System Summary` 表。
   - 新增 `System Scenarios` section，展示每个 strategy 在每个 episode/scenario 下的：
     - planner skills
     - success
     - executor error
     - replanned skills
   - report 仍只读取 JSON，不重新运行 PyBullet 或 mock system。

4. 增加测试与文档
   - 新增 `tests/test_causal_explore_system_eval.py`。
   - 扩展 `tests/test_causal_explore_eval.py` 和 `tests/test_causal_explore_report.py`。
   - 更新 `embodied_agent/README.md`，明确 artifact-level、planner-level、system-level 三层评估。
   - 更新 `embodied_agent/CHECKPOINTS.md`，新增 `CP-2026-04-27-causal-explore-system-eval-v1`。

```bash
python run_mock_system_demo.py --catalog-path outputs/causal_explore/causal_system_eval_v1_smoke/curiosity/catalog.json --scenario red_place_failure
```

结果：

- `evaluation_summary.json`、`planner_eval_summary.json`、`system_eval_summary.json`、`evaluation_report.md` 均已生成。
- 三个 strategy 各有 `system_metrics.jsonl`。
- `red_place_failure` 能记录 executor error：`released_outside_zone`。
- `blue_success` 能记录成功执行。

Manual smoke 关键指标：

| Strategy | System cases | Success rate | Failures | Replans | Probe steps |
| --- | ---: | ---: | ---: | ---: | ---: |
| random | 6 | 0.5000 | 3 | 3 | 0 |
| curiosity | 6 | 0.5000 | 3 | 3 | 1 |
| causal | 6 | 0.5000 | 3 | 3 | 0 |

解释：

- 当前 system eval v1 的两个固定 scenario 中，`blue_success` 是成功路径，`red_place_failure` 是故意注入 place failure 的恢复路径，所以 3 episodes 下每个 strategy 是 3 成功 + 3 失败恢复 case。
- `curiosity` 在其中一个 red failure case 中触发了 `probe`，并在 replanned path 中保留 `probe`，说明 CausalExplore 的不确定性信息已经能够影响 planner/system trace。

## 对 `EMBODIED_AGENT_TEAM_DIVISION.md` 的贡献

这份分工文档定义了项目的三层定位：

- 成员 A：强化学习运动控制与仿真环境，即“小脑”。
- 成员 B：CausalExplore 建模与物体理解，即“认知桥梁”。
- 成员 C：大模型任务规划与系统集成，即“大脑”。

今日工作的主要贡献在 B/C 交界，同时也使用了 A 的 executor 返回结构作为 system-level 事实来源。

具体贡献：

1. 对成员 B 的贡献：CausalExplore 输出不再只是 artifact 或研究者可读结果
   - 分工文档要求 CausalExplore 输出对象属性、用途候选、不确定性、推荐探索动作、接触区域和技能约束，并且这些结果要能被规划模块直接读取。
   - 今日 system eval 通过 episode catalog 驱动 mock planner/system demo，证明 CausalExplore catalog 已经可以作为下游系统输入，而不是只停留在离线 artifact。

2. 对成员 C 的贡献：规划与系统集成有了 closed-loop 对比表
   - 分工文档要求成员 C 负责规划-执行映射、失败反馈、重规划机制和系统演示材料。
   - 今日新增的 `System Summary` 和 `System Scenarios` 把 planner skills、executor success/error、replanned skills 放到同一张 report 中，直接服务系统演示和答辩叙事。

3. 对成员 A/C 接口的贡献：executor 失败反馈进入评估事实链
   - 分工文档要求执行层返回 `success`、`reward`、`contact_state`、`final_state`、`error_code`、`rollout_path`。
   - 今日 system eval 使用 `executor_result.error_code = released_outside_zone` 和 `replanned_steps` 作为统计字段，说明执行层失败码已经能被上层评估和报告消费。

4. 对协作边界的贡献：没有让 LLM 或 planner 越界控制连续动作
   - 分工文档强调大模型不直接输出连续动作，CausalExplore 不替代技能执行，技能层不负责用途推断。
   - 今日实现严格保留结构化 planner step、mock executor 和 CausalExplore artifact catalog 的边界，符合“大脑-认知桥梁-小脑”的职责划分。

当前不足：

- 仍是 mock system-level evaluation，不是真实 PyBullet executor batch，也不是 MuJoCo/IsaacLab 在线探索闭环。
- 对象仍是 red/blue block，尚未覆盖分工文档中建议的鼠标、按钮、抽屉、门、旋钮等交互对象。
- 指标还没有属性准确率、用途准确率、平均探索步数等成员 B 的核心模型指标。

## 今日结论

今天的核心价值不是提升 pick/place 技能本身，而是补齐了 CausalExplore 评估链路中的 system-level 层。

原先链路大致是：

```text
probe artifact -> planner 是否插入 probe -> report
```

现在推进为：

```text
probe artifact
  -> planner-level probe behavior
  -> mock system success/failure/replan behavior
  -> evaluation_summary + system_eval_summary + evaluation_report
```

这对项目路线的意义是：

- 对分工文档：让成员 B 的 CausalExplore 输出真正进入成员 C 的系统集成评估，并复用成员 A 的 executor result schema。
- 对 NewPlan：推进了 P0 的 checkpoint 固化、P2 的接口适配、P3 的 mock closed-loop integration，并为 P4 的实验表格打下 JSON/report 基础。

## 下一步建议

1. 将 system eval 从 mock executor 扩展到真实 PyBullet executor batch，至少先做小规模 deterministic smoke。
2. 增加 planner 是否使用 uncertainty 的 ablation，例如禁用 CausalExplore provider 后对比 planner/system trace。
3. 扩展 scenario，不只包含 `blue_success` 和 `red_place_failure`，加入更多对象、更多失败类型和需要主动 probe 的任务。
4. 为 P2 补充真正的 simulator-backed CausalExplore 指标：
   - 属性推断准确率
   - 用途判断准确率
   - 平均探索步数
   - uncertainty 下降曲线
5. 将当前 `evaluation_report.md` 的 system summary 表整理成后续论文/答辩可直接引用的实验表模板。
