# 飞书 AI 大创文档对齐总结

> 2026-05-02 状态提示：本文反映 2026-04-19/20 的路线对齐。当前项目已完成 MuJoCo simulator-backed v1 迁移，因此“还没有 MuJoCo 对照验证”“基于 PyBullet 的 baseline”已更新为“MuJoCo 默认验证后端，PyBullet legacy，IsaacLab 待接入”。

## 使用到的飞书文档

本次整理参考了飞书知识库 `AI / 大创` 下的 3 份核心文档：

- `项目调研`
- `对比表`
- `实现路线`

## 飞书文档给出的核心路线

飞书文档的建议路线比较明确：

- 高层由大模型负责任务理解、任务分解和子任务排序。
- 中层需要结构化执行表示，例如 JSON plan、状态机或行为树。
- 低层由 RL 技能库负责 reach、grasp、lift、place 等连续控制。
- 更长期的平台建议是 Isaac Lab / Isaac Sim。
- MuJoCo 更适合作为 sim2sim 验证和 benchmark 对照平台。
- 真机阶段优先考虑 Franka Panda。

## 飞书文档不建议的路线

飞书文档不建议一开始就做这些事：

- 不建议一开始做 LLM 直接端到端控制机械臂。
- 不建议一开始就做长程复杂任务。
- 不建议跳过技能拆分，直接训练完整复杂任务。

## 当前本地项目和飞书路线的重合部分

### 已经对齐的部分

- 已经是分层结构，而不是端到端一体化黑盒。
- 已经有高层规划器、执行器、技能库、仿真环境这几层接口。
- 已经把任务拆成 `pick` 和 `place` 两个技能，而不是直接训练整条链。
- 已经有训练、评估、测试和实验归档机制。

### 还没对齐的部分

- 高层还不是大模型，而是规则规划器。
- 中层还不是行为树或更完整的结构化任务图。
- 低层虽然已经有 RL 探索，但主线尚未稳定形成纯 learned full chain。
- 仿真平台还没有切到 Isaac Lab / Isaac Sim。
- 历史判断：当时还没有 MuJoCo 对照验证；2026-05-02 后 MuJoCo 已成为默认 simulator-backed 验证后端。
- 还没有 Franka 真机接口。
- 还没有视觉感知闭环。

## 从飞书路线回看当前项目的真实定位

更准确的定位应该是：

- 当前项目已经做出了“分层式具身系统的第一版骨架”。
- 历史判断：当时主要完成了“脚本/学习技能 + 规则规划 + PyBullet 仿真”的基础闭环；当前已升级为 MuJoCo simulator-backed v1。
- 它是后续更完整路线的前置 baseline，而不是最终系统。

## 飞书路线对当前决策的启发

如果按飞书路线推进，当时最合理的选择不是马上迁移平台，而是先补齐以下内容；2026-05-02 后已完成 MuJoCo simulator-backed v1 迁移，下面列表保留为历史决策依据：

1. 让当前技能库真正稳定。
2. 统一评估口径，把 scripted、BC、RL 对比讲清楚。
3. 在现有技能接口之上接一个结构化 LLM planner。
4. 再视资源决定是先切 Isaac Lab，还是先补 MuJoCo 对照环境。

## 当前阶段的建议表述

如果之后要写汇报或答辩，比较稳妥的表述是：

- 已完成早期 PyBullet 具身操作 baseline，并在 2026-05-02 迁移到 MuJoCo simulator-backed v1。
- 已实现任务规划、技能执行、训练评估闭环。
- 已验证脚本基线、行为克隆和纯 RL 的可行性。
- 下一阶段将补齐高层 LLM 规划、感知层和更强仿真平台路线。

## 当前推荐的短期策略

结论很简单：

- 飞书路线是对的。
- 当前本地项目已经走在这条路上，但还处于第一阶段。
- 所以当时最该做的是“补强 baseline”，而不是“立刻大迁移”；当前已进入 MuJoCo v1 验证后端和 IsaacLab 接入准备阶段。
