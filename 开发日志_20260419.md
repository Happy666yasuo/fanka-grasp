# 开发日志 — 2026年4月19日

## 本轮主线

这轮工作的重点已经从“把单物体 PyBullet 主线跑通”转成“把 learned pick/place 放到多物体 benchmark 里做真实压力测试，再基于失败模式回推环境语义和训练分布问题”。

本轮实际完成的核心事情有四条：

- 把 failure_history 做成可直接汇总的失败分析工具，能输出恢复统计和失败表。
- 把 `projects/embodied_agent` 扩展成真正可跑的多物体 benchmark，而不是只看单物体 smoke。
- 修正 learned place 的 release 语义，不再在任意 `release_event` 上立刻结束，而是保留几步 settling 再按最终落点评分。
- 对 learned pick 做训练分布和运行时分布核对，并尝试了新的 SAC/BC 路线，但目前仍没有拿到比旧 conservative baseline 更强的多物体 pick manifest。

## 今日已完成工作

### 1. 失败分析工具已落地

- 扩展了 `src/embodied_agent/failure_analysis.py`，把 pick/place 失败类别和 recovery 信息统一整理。
- 新增 `analyze_recovery_failures.py`，可以直接读取 evaluation run 目录，把 failure_history 汇总成 Markdown/JSON 表。
- 这一步的意义不是多了一个脚本，而是之后所有多物体 smoke 的判断不再只看 success_rate，而是能拆出“失败发生在 pick 还是 place、是否触发重规划、是否恢复成功”。

### 2. 多物体 benchmark 第一批已经完整接通

- `evaluation.py`、`planner.py`、`executor.py`、`skills.py`、`simulator.py` 这一整条链已经支持 `task_pool`、显式 `object_name/zone_name` 和多实体场景。
- 多物体布局采样已经从 rejection sampling 改成 candidate slots，避免 3 个物体和 3 个区域同时激活时频繁采样失败。
- 新增 `configs/eval/batch_eval_12_multi_object_smoke.yaml` 和对应测试覆盖。
- 脚本基线在多物体 smoke 上保持 `1.00`，说明 benchmark 本身和任务解析链是可用的。

### 3. learned skill 的多物体首轮结果和失败诊断已经明确

首轮多物体 hybrid smoke 的结果是：

| 实验 | 成功率 |
|------|--------|
| scripted_multi_object_smoke | 1.00 |
| learned_pick_multi_object_smoke | 0.50 |
| learned_place_multi_object_smoke | 0.50 |
| learned_pick_place_multi_object_smoke | 0.25 |

这组结果很关键，因为它说明：

- 系统不是“多物体 benchmark 本身坏了”，因为 scripted 还是 12/12。
- learned pick 和 learned place 都会在多物体里掉点，但掉点机理不同。

失败分析的结论是：

- learned pick 主要卡在 pre-grasp approach / acquisition，表现为重复 `execution_error`，且 recovery 基本救不回来。
- learned place 主要是 `released_outside_zone`，说明它不是不会 release，而是 release 时机和最终落点控制不对。

### 4. place 训练语义已经修正，并完成回归验证

这轮对 `src/embodied_agent/rl_envs.py` 做了关键修正：

- place 不再在任意 `release_event` 上立刻终止。
- 环境新增 `post_release_settle_steps`，默认给 place 保留几步 settling 窗口。
- 最终 reward 和 success/penalty 改为按 settled landing state 计算，而不是按松爪瞬间计算。

这一步已经通过两层验证：

- `tests/test_rl_envs.py` 增加了 settling 语义测试并通过。
- `configs/eval/place_formal_eval.yaml` 回归结果保持很强：

| 实验 | 成功率 |
|------|--------|
| place_fixed_50 | 1.00 |
| place_random_50 | 1.00 |

所以当前结论很明确：

- place 的单技能模型并没有被新语义打坏。
- full-task 多物体里的 place 问题，更像是“训练时 staging-to-zone 假设”和“真实 post-pick runtime 状态”不一致，而不是单纯 reward 语义错了。

### 5. pick 的训练分布核对和新一轮重训已经做完，但还没找到更优 manifest

这轮先做了输入分布核对，再做训练尝试。

分布核对的结论是：

- 当前主力 pick manifest 虽然在单技能 formal eval 上依然很强，但它的训练覆盖基本还是单布局附近。
- 多物体首轮 pick 失败主要不是 target routing 错了，而是初始 `ee_object_distance`、EE 横向位置和 staging 后高度分布已经偏离原训练流形。
- 最明显的偏移是 lateral staging pose。训练期 `ee_y` 在正值窄区间内，而多物体首轮尝试几乎全部落到负 `ee_y` 一侧。

随后做了两条新路线：

| 路线 | 结果 |
|------|------|
| SAC broad-uniform coverage | best_success_rate 0.50，后续塌到 0.00 |
| BC candidate-grid v2 | best_success_rate 0.45，后续过拟合回落 |

然后把 BC pick v2 接回多物体 smoke，结果是：

| 实验 | 成功率 |
|------|--------|
| scripted_multi_object_smoke | 1.00 |
| bc_pick_candidate_grid_multi_object_smoke | 0.25 |
| learned_place_multi_object_smoke | 0.50 |
| bc_pick_place_multi_object_smoke | 0.083 |

最后又把旧 conservative SAC pick 单独放回多物体 smoke，对比结果是：

| 实验 | 成功率 |
|------|--------|
| conservative_pick_multi_object_smoke | 0.50 |

因此到目前为止，结论不是“新模型还差一点”，而是：

- 现有 coverage 扩展办法没有把 pick 上限推高。
- 当前没有一个比旧 conservative pick 更好的 ready-made 多物体 pick manifest。

## 当前阶段判断

### 已经确认的事实

- scripted 多物体基线可靠。
- pick 单技能 formal eval 仍强，现有主力 pick manifest 在单技能上是 `0.94 / 0.92`。
- place 单技能 formal eval 在新 settling 语义下仍是 `1.00 / 1.00`。

### 当前真正的瓶颈

- pick 的问题更像 representation / objective / runtime manifold mismatch，而不是简单的 task routing 错误。
- place 的问题更像训练状态和真实 post-pick 状态之间的 runtime mismatch，而不是 release 语义本身。

### 这轮最重要的收获

这轮最值钱的成果不是又多训出一个模型，而是把问题边界收紧了：

- place 的环境语义已经理顺。
- pick 和 place 在多物体里的失败机理已经被拆开。
- 下一步该改什么，不再模糊。

## 下一步应该做的工作

### P0：先解决 learned place 的 runtime mismatch

1. 采样真实 `post-pick` 状态，和当前 `use_staging=true` 的 place 训练起始状态做分布对比。
2. 新增“从真实 post-pick 状态起步”的 place 评估配置，确认 learned place 为什么单技能强、全链路弱。
3. 如果差异明显，就补一版 place 训练：从 post-pick state 或更贴近多物体 runtime 的 staged state 起步，而不是只从理想 staging pose 起步。
4. place 的阶段性验收标准不是再看单技能 1.00，而是把 `learned_place_multi_object_smoke` 从 `0.50` 往上推，并把 place 主导的 recovery failure 明显压下去。

### P0：再做 pick 的 representation / objective 改造

1. 不再继续做“只加布局随机化”的同类试验，因为这条路线已经证明收益有限。
2. 优先考虑更强的相对量表征和对 staging 几何偏移更鲁棒的输入设计，减少对绝对 `object/zone` 坐标和单一 EE 起始位姿的依赖。
3. 用更聚焦的 expert 数据或 BC/SAC 混合目标，专门覆盖当前多物体里最容易失败的 lateral approach 区域，而不是继续做宽而浅的全空间随机化。
4. pick 的阶段性验收标准是先超过当前 `0.50` 多物体 smoke 上限，再看 full hybrid 是否改善。

### P1：等 pick 和 place 都换新结果后，再回跑 full hybrid

等 learned place 和 learned pick 都有新的 runtime-aligned manifest 后，再回跑：

- `configs/eval/batch_eval_12_multi_object_hybrid_smoke.yaml`
- 对应 failure analysis 汇总表

这一步才是检验“多物体 benchmark 这一轮是否真正收敛”的最终验收。

### P1：同步答辩和项目文档

当前最值得写进答辩材料的不是“我们又训练了多少轮”，而是：

- 已经完成了从单物体到多物体 benchmark 的工程升级。
- 已经建立 failure-driven 诊断链，而不是只看 success_rate。
- 已经明确 pick 和 place 的失败来自两类不同问题，后续路线是可解释的。
