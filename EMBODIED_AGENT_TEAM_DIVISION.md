# 项目分工文档：基于强化学习运动控制和大模型任务规划的具身智能体研究

## 1. 文档目的

这份文档用于明确本项目的技术定位、三人分工、阶段目标、接口边界和验收标准，避免后续出现“每个人都在做一点，但没有人对闭环结果负责”的问题。

本项目的核心定位不是单独做强化学习，也不是单独做大模型，而是构建一个“可解释、可执行、可验证”的具身智能体系统：

- 强化学习运动控制负责低层动作执行，承担“小脑”角色。
- 大模型任务规划负责高层任务理解、目标拆解和步骤编排，承担“大脑”角色。
- CausalExplore 负责环境中的物体理解、因果属性推断和用途判断，承担“大脑与小脑之间的认知桥梁”角色。

换句话说，CausalExplore 不是机械臂控制器，也不是最终任务规划器，而是“让系统知道当前物体是什么、可能怎么用、为什么应该这样操作”的关键中间层。

## 2. 项目总目标

### 2.1 总体目标

围绕“基于强化学习运动控制和大模型任务规划的具身智能体研究”，搭建一个从感知、理解、规划到执行的完整仿真闭环，使机器人能够：

1. 看见或接触到环境中的物体。
2. 通过 CausalExplore 主动推断物体的物理属性、交互方式和可操作用途。
3. 由大模型根据任务目标和物体性质生成合理的高层任务步骤。
4. 由强化学习技能策略执行低层动作并反馈执行结果。
5. 在执行失败或信息不足时，重新触发探索、修正推断和调整计划。

### 2.2 论文与演示目标

项目最终至少应交付以下两类结果：

1. 研究结果
   - 证明 CausalExplore 提供的物体属性与用途信息，能够显著改善大模型任务规划质量或执行成功率。
   - 证明强化学习技能策略能够稳定承担低层执行，不需要大模型直接输出连续控制量。

2. 系统演示
   - 在仿真环境中完成“目标输入 -> 任务规划 -> 主动探索 -> 属性推断 -> 技能执行 -> 成功/失败反馈”的完整流程。
   - 以鼠标、按钮、抽屉、门、旋钮等对象为代表，展示机器人如何根据观察和交互推断合理用法。

## 3. 系统定位：大脑、认知桥梁、小脑

## 3.1 大脑：大模型任务规划层

大模型的职责是解决“做什么、先做什么、后做什么”的问题，而不是“每个关节转多少度”的问题。

它负责：

- 读取任务指令和环境上下文。
- 根据 CausalExplore 提供的物体属性和用途候选，生成任务步骤。
- 判断当前步骤应调用哪类技能，例如抓取、按压、推动、拉动、旋转、拖动。
- 在执行失败时根据反馈进行重规划。

它不负责：

- 直接输出连续动作控制。
- 代替传感器或交互模型推断物理属性。
- 代替强化学习策略完成精细动作。

## 3.2 认知桥梁：CausalExplore 环境理解层

CausalExplore 解决的是“这个东西可能是什么、具有什么物理属性、应该如何操作”的问题。

它负责：

- 结合视觉、接触、力觉等信息推断潜在属性。
- 通过主动交互降低对物体性质的不确定性。
- 形成面向规划层可用的用途表达，例如“可按压”“可拖动”“可旋转”“需要抓握后移动”。
- 输出与下游技能选择相关的约束信息，例如接触点、交互方向、置信度和探索建议。

它不负责：

- 独立完成全局任务规划。
- 直接输出稳定的连续控制轨迹。
- 代替技能策略完成复杂接触控制。

## 3.3 小脑：强化学习运动控制层

强化学习层解决的是“如何稳定、快速、鲁棒地完成动作”的问题。

它负责：

- 学习和执行底层技能策略。
- 接收规划层或中间层给出的技能调用指令。
- 处理接触控制、轨迹跟踪、末端执行器姿态和力控制。
- 输出动作执行结果和状态反馈。

它不负责：

- 独立推断物体用途。
- 独立完成任务分解。
- 代替 CausalExplore 解释为什么当前对象应该采用某种操作方式。

## 4. 总体技术路线

建议采用“先 MuJoCo 跑认知闭环，再 IsaacLab 强化技能，再做集成”的路线，而不是一开始同时推进所有平台。

### 2026-05-02 当前进度更新

这条路线已经完成第一步的工程迁移：

- MuJoCo 已成为当前 simulator-backed 测试和 CausalExplore probe 的默认后端。
- PyBullet 仅作为 legacy backend 保留。
- `embodied_agent` 已新增 `PickPlaceSimulationProtocol` 和 `create_pick_place_simulation(backend="mujoco")`。
- Ralph Phase1/Phase3 的 simulator-backed 路径已切到 MuJoCo，并覆盖红/绿、蓝/黄、黄/蓝三类目标组合。
- 当前验证结果：`embodied_agent 112 OK`，Ralph Phase1 `64 OK`，Phase2 `88 OK`，Phase3 `65 OK`。

后续分工应按这个状态继续推进：

- 成员 A：接入 IsaacLab 技能训练 adapter，或提升 MuJoCo backend 到更真实的 robot dynamics/controller。
- 成员 B：基于 MuJoCo probe 刷新 CausalExplore 正式实验和图表。
- 成员 C：保持结构化 planner/replan 口径，刷新论文材料中所有实验数值和 backend 描述。

### 阶段 0：问题定义与对象集合设计

目标：定义任务、对象类别、属性标签和用途标签。

输出：

- 最小对象集。
- 属性标签表。
- 用途标签表。
- 下游任务清单。

### 阶段 1：CausalExplore 最小闭环验证

目标：在 MuJoCo 中实现“观察/接触 -> 主动探索 -> 属性推断 -> 用途判断”。

输出：

- 可运行的探索环境。
- 主动探索动作原语。
- 属性推断模型。
- 用途推断结果。

### 阶段 2：强化学习技能库建设

目标：在 IsaacLab 或等价仿真平台训练可复用的技能策略。

- 按压技能。

- 旋转技能。
- 技能调用接口。

目标：把 CausalExplore 输出作为大模型规划输入的一部分，形成“理解支持规划”的闭环。

输出：

- 任务规划提示模板。
- 结构化规划输出格式。
- 规划到技能的映射器。
- 失败反馈和重规划流程。

### 阶段 4：评测、消融和演示

目标：证明各模块贡献，输出论文与答辩材料。

输出：

- 主实验表格。
- 消融实验。
- 视频演示。
- 中期或结题汇报材料。

## 5. 三人分工总览

建议把三位成员的角色固定为“环境与技能”“因果认知”“规划与集成”，每个人有明确主责，同时保留一个协作副责，避免单点故障。

| 成员角色 | 主责方向 | 副责方向 | 核心目标 |
| --- | --- | --- | --- |
| 成员 A | 强化学习运动控制与仿真环境 | 系统部署与演示支持 | 把动作执行层做稳，保证技能可调用、可复现、可评测 |
| 成员 B | CausalExplore 建模与物体理解 | 数据标注与实验分析 | 把环境理解层做对，保证属性推断和用途判断有效 |
| 成员 C | 大模型任务规划与系统集成 | 文档、论文与答辩统筹 | 把高层规划与整条链路串起来，保证系统闭环能展示 |

## 6. 详细分工

## 6.1 成员 A：强化学习运动控制与仿真环境负责人

### 主责定位

成员 A 对“小脑”负责，目标是让机器人拥有稳定、可调用、可组合的动作技能，并提供可供上层调用的仿真执行接口。

### 主要工作内容

1. 搭建和维护仿真环境。
2. 定义机器人控制接口、动作空间、观测空间和任务终止条件。
3. 为抓取、按压、推动、拉动、旋转、拖动等动作训练或实现技能策略。
4. 统一技能调用方式，使上层规划器能够通过标准接口触发技能。
5. 记录执行结果、轨迹日志、接触信息和成功判定。

### 必做子任务

1. 环境搭建
   - 建立 MuJoCo 对象交互环境。
   - 若采用 IsaacLab 训练技能，需要同步建立 IsaacLab 技能训练环境。
   - 对象模型、碰撞体、关节、质量、摩擦参数要有统一配置文件。

2. 技能策略
   - 优先做基础技能：reach、grasp、press、push、pull、rotate。
   - 每个技能都要定义输入参数、成功条件和失败码。
   - 每个技能都要能被脚本独立测试，而不是只能在完整系统里调通。

3. 执行层接口
   - 提供统一的 skill executor。
   - 提供标准返回值，包括 success、reward、contact_state、final_pose、error_code。
   - 支持被大模型规划模块和 CausalExplore 模块共同调用。

### 关键交付物

- 环境配置说明文档。
- 技能训练脚本与评测脚本。
- 技能库清单。
- 执行日志导出工具。
- 技能成功率统计图。

### 验收标准

1. 至少 4 个基础技能可重复调用。
2. 每个技能在固定对象上达到可接受成功率。
3. 技能接口文档齐全，其他成员能无障碍调用。
4. 仿真重跑后结果波动在可解释范围内。

### 风险点

- 技能训练周期过长。
- 仿真平台切换导致接口频繁变化。
- 技能执行效果好，但缺少统一日志和评测口径。

### 风险应对

- 先用规则控制器或脚本策略占位，保证系统先闭环。
- 强化学习策略成熟后再替换占位技能。
- 所有技能统一走一个执行器接口，避免重复适配。

## 6.2 成员 B：CausalExplore 建模与物体理解负责人

### 主责定位

成员 B 对“认知桥梁”负责，目标是让系统在面对新物体时，不是盲目执行，而是先理解物体的性质、潜在交互方式和用途，再把结构化结果提供给上层规划器。

### 主要工作内容

1. 设计最小对象集和属性标签体系。
2. 实现 CausalExplore 的探索、属性推断和用途表达模块。
3. 建立从视觉/接触输入到物体性质推断的训练与评测流程。
4. 做主动探索与被动观察的对比实验。
5. 输出供大模型使用的结构化物体理解结果。

### 必做子任务

1. 对象与标签体系设计
   - 物体类别至少覆盖刚体、关节体、可变形体、复合体中的若干代表对象。
   - 属性标签至少包括质量、摩擦、关节类型、阻尼、可按压性、可旋转性、可拖动性。
   - 用途标签要能直接服务规划层，例如“需要抓握后操作”“适合平面拖动”“可按压触发状态变化”。

2. CausalExplore 最小实现
   - 实现主动探索动作选择逻辑。
   - 实现多模态属性推断模型。
   - 实现置信度或不确定性估计。
   - 实现用途推断头或用途映射规则。

3. 数据与评测
   - 采集探索轨迹和属性标签。
   - 构建训练集、验证集、测试集。
   - 输出属性准确率、用途判断准确率、平均探索步数等指标。

4. 对规划层的输出格式设计
   - 输出对象类别猜测。
   - 输出属性后验分布或高置信属性。
   - 输出候选用途列表。
   - 输出建议交互方式和推荐接触点。

### 关键交付物

- 对象与标签规范。
- CausalExplore 模型代码。
- 训练与评测脚本。
- 物体理解结果样例集。
- 主动探索与随机探索对比图。

### 验收标准

1. 至少在最小对象集上实现稳定的属性推断。
2. 主动探索明显优于随机探索或纯视觉基线。
3. 输出结果可以直接被规划模块读取，而不是只对研究者可读。
4. 至少有 1 个典型案例能展示“先看不确定，经过交互后再确认用途”。

### 风险点

- 属性体系定义太多，导致标注和训练失控。
- 模型输出和规划层需求脱节。
- 只做了属性分类，没有做出面向任务的用途表达。

### 风险应对

- 先做 6 到 10 个对象的最小集合。
- 先保留少量关键属性，不追求一步到位覆盖所有物理量。
- 所有输出优先服务规划层和技能层，而不是只追求模型论文指标。

## 6.3 成员 C：大模型任务规划与系统集成负责人

### 主责定位

成员 C 对“大脑”和整体闭环负责，目标是让任务指令、环境理解结果和技能执行结果真正连成系统，而不是三个孤立模块。

### 主要工作内容

1. 设计任务输入、规划提示模板和结构化输出格式。
2. 把 CausalExplore 输出转换成大模型可利用的环境上下文。
3. 建立规划结果到技能调用的映射流程。
4. 设计失败反馈、异常状态和重规划机制。
5. 负责系统演示脚本、文档、论文主线和答辩叙事。

### 必做子任务

1. 大模型规划接口
   - 定义任务输入格式。
   - 定义上下文输入格式，包括对象属性、用途候选、置信度和执行约束。
   - 定义结构化输出，例如 step list、skill name、target object、precondition、expected outcome。

2. 规划到执行的映射
   - 把规划输出映射到技能库调用。
   - 对技能前提条件做检查，例如是否已经抓取、是否接触到目标、是否确定了对象用途。
   - 若条件不足，触发补充探索而不是盲目执行。

3. 重规划机制
   - 技能失败时回收失败原因。
   - 区分“技能没执行好”和“规划本身不合理”。
   - 当 CausalExplore 不确定度过高时，优先要求系统继续探索。

4. 系统演示与研究材料
   - 负责最终 demo 路线设计。
   - 负责视频脚本、汇报结构、图表汇总。
   - 负责阶段性周报和对外表达材料。

### 关键交付物

- 大模型提示模板。
- 规划结果 JSON 或等价结构化格式。
- 规划-执行映射器。
- 演示主脚本。
- 论文或汇报主线草稿。

### 验收标准

1. 给定任务目标后，大模型能输出结构化步骤，而不是自由文本描述。
2. 规划模块能利用 CausalExplore 输出做出比纯视觉描述更合理的决策。
3. 出现执行失败时，系统能给出继续探索、重试技能或调整任务步骤中的至少一种响应。
4. 最终闭环演示可稳定复现。

### 风险点

- 大模型提示过于宽泛，输出不稳定。
- 规划结果与技能接口字段不一致。
- 过度依赖大模型文本能力，导致系统不可控。

### 风险应对

- 所有规划输出都采用结构化 schema。
- 限制大模型只做任务分解和技能选择，不直接控制关节。
- 为常见对象和任务准备少量手工规则兜底。

## 7. 协作边界与接口约定

为了避免集成时互相等待，三位成员必须尽早约定输入输出格式。

### 7.1 CausalExplore 输出给规划层的最小字段

建议至少包含以下内容：

| 字段 | 含义 |
| --- | --- |
| object_id | 当前交互对象唯一标识 |
| object_category | 物体类别猜测 |
| property_belief | 关键属性后验或置信度 |
| affordance_candidates | 候选用途列表 |
| uncertainty_score | 当前不确定度 |
| recommended_probe | 建议补充探索动作 |
| contact_region | 推荐接触区域或关键点 |
| skill_constraints | 对技能调用的约束条件 |

### 7.2 规划层输出给执行层的最小字段

| 字段 | 含义 |
| --- | --- |
| task_id | 当前任务编号 |
| step_index | 当前步骤序号 |
| target_object | 目标对象 |
| selected_skill | 要调用的技能 |
| skill_args | 技能参数 |
| preconditions | 执行前提 |
| expected_effect | 预期结果 |
| fallback_action | 失败后的回退策略 |

### 7.3 执行层返回给上层的最小字段

| 字段 | 含义 |
| --- | --- |
| success | 是否成功 |
| reward | 执行回报 |
| final_state | 执行后的关键状态 |
| contact_state | 接触状态或接触变化 |
| error_code | 失败类型 |
| rollout_path | 轨迹或日志位置 |

下面给出一个适合三人团队的 8 周执行框架，可根据课程节奏或导师节点微调。

| 周次 | 阶段目标 | 成员 A | 成员 B | 成员 C |

| 第 3 周 | 建立最小 CausalExplore 原型 | 提供探索动作原语 | 完成属性推断基线 | 接入环境上下文给规划器 |
| 第 4 周 | 完成主动探索闭环 | 支持交互执行回放 | 完成主动探索与随机探索对比 | 完成规划到技能映射初版 |
| 第 5 周 | 建立 RL 技能库第一版 | 完成 2 到 4 个技能策略 | 输出用途候选和约束字段 | 调通结构化规划调用技能 |
| 第 6 周 | 完成系统集成 | 提升技能稳定性 | 提升用途推断稳定性 | 完成失败反馈与重规划 |
| 第 7 周 | 完成主实验和消融 | 输出技能成功率 | 输出属性和用途指标 | 输出规划质量和整体成功率 |
| 第 8 周 | 完成演示与文档 | 准备环境演示脚本 | 准备认知可视化结果 | 汇总论文、PPT、视频和总报告 |

## 9. 共同任务与公共责任

虽然三人各有主责，但以下内容必须共同承担：

1. 统一实验命名、日志命名和目录结构。
2. 每周同步当前接口是否发生变化。
3. 每次实验必须记录版本、参数、对象、任务、成功率和异常情况。
4. 每周至少进行一次联调，而不是各自只跑自己的模块。
5. 任何对公共接口的修改必须提前通知其他成员。

## 10. 推荐协作机制

## 10.1 周会机制

建议每周固定一次正式周会和一次短会：

1. 正式周会
   - 汇报本周完成内容。
   - 展示可运行结果而不是只讲计划。
   - 明确下周依赖关系和风险。

2. 联调短会
   - 只处理接口联调和系统闭环问题。
   - 保证“大脑-桥梁-小脑”链路不断。

## 10.2 版本管理机制

建议统一采用以下原则：

1. 每个成员维护自己的模块目录。
2. 公共接口文件由成员 C 牵头统一维护，成员 A、B 共同确认。
3. 所有实验结果写入统一实验日志。
4. 所有可复现实验必须附带运行命令和参数说明。

## 10.3 文档责任机制

建议文档分工如下：

- 成员 A 负责环境与技能文档。
- 成员 B 负责模型、数据和指标文档。
- 成员 C 负责系统总文档、演示材料和论文主线。

## 11. 评价指标与验收口径

项目不应只看一个总成功率，而应至少从以下几个维度评估：

1. 物体属性推断准确率。
2. 用途推断准确率。
3. 平均探索步数。
4. 任务规划合理率。
5. 技能执行成功率。
6. 端到端任务成功率。
7. 新物体上的泛化表现。
8. 失败后重规划成功率。

对应到三位成员：

- 成员 A 重点负责技能执行成功率和环境稳定性。
- 成员 B 重点负责属性与用途推断指标。
- 成员 C 重点负责规划合理率和端到端任务成功率。

## 12. 推荐的最小演示案例

为了让项目尽快形成能展示的结果，建议先做一个最小但完整的代表性案例。

### 案例：鼠标用途推断与操作

1. 机器人观察桌面上的鼠标。
2. CausalExplore 输出初步结论：该物体适合抓握、平面拖动、顶部按压、可能具有滚动部件。
3. 若当前不确定度较高，机器人进行轻触、按压或拖动试探。
4. 系统更新对“点击”“拖动”“滚轮操作”的用途判断。
5. 大模型根据任务目标，例如“把鼠标移到指定区域并执行一次点击”，调用合适技能序列。
6. 强化学习技能层完成抓取、放置、拖动、按压动作。

这个案例能够同时体现：

- CausalExplore 在规划前提供环境理解。
- 大模型不是盲目规划，而是利用物体性质做任务分解。
- 强化学习策略承担真实动作执行。

## 13. 最终建议

这个项目最容易出现的偏差有两个：

1. 把大模型做成一个“什么都管”的黑盒。
2. 把强化学习做成一个“只要训练够久就能解决所有问题”的黑盒。

正确的做法是：

- 让大模型负责高层任务理解与步骤安排。
- 让 CausalExplore 负责物体理解、用途推断和探索决策支持。
- 让强化学习负责低层技能执行。

只有这样，“大脑-认知桥梁-小脑”的整体结构才清晰，实验结果也更容易解释、复现和答辩。
