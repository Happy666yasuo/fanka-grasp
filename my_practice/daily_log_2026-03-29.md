### 4. Phase 3: Reward Shaping — 三种奖励策略 (CP-3)

**目标**: 实现 shaped / PBRS / curriculum 三种奖励策略，支持训练时切换。

- 扩展 `envs/mdp/rewards.py`: 新增 3 个高级奖励函数
  - **`shaped_multi_stage`**: 4 阶段密集奖励
    - r = 1.0·r_reach + 3.0·r_grasp + 5.0·r_lift + 8.0·r_hold
    - 抓取代理信号: EE 距离 < 阈值 + gripper 闭合
  - **`pbrs_shaping`**: Potential-Based Reward Shaping (Ng et al., 1999)
    - Φ(s) = -(‖p_ee - p_obj‖ + |z_obj - z_target|)
    - F(s, s') = γΦ(s') - Φ(s)，保证最优策略不变
    - 实现了 `env._pbrs_prev_potential` 状态跟踪 + reset 检测
  - **`curriculum_reward`**: 课程学习奖励
    - Easy (< 50K steps): 重 reach + grasp
    - Medium (50K~200K): reach + grasp + lift
    - Hard (> 200K): 仅 lift + hold
- 扩展 `envs/franka_grasp_env_cfg.py`: 新增 3 种 RewardsCfg
  - `SparseRewardsCfg`: 纯稀疏 (消融基线)
  - `ShapedRewardsCfg`: shaped_multi_stage + penalties
  - `PBRSRewardsCfg`: sparse lift + PBRS + penalties
- 更新 `scripts/train.py`: 实现 `--reward_type` 配置切换
  - reward_map 字典: sparse → SparseRewardsCfg, shaped → ShapedRewardsCfg, pbrs → PBRSRewardsCfg
- 新建 `scripts/check_reward_range.py`: 奖励范围诊断工具
  - 随机动作跑 N 步, 报告 min/max/mean/std
  - 逐 reward term 细分报告 (使用 `reward_manager._step_reward`)
  - NaN/Inf 检测 + 幅值健康度评估
- **验证**:

## 💡 经验与收获

1. **AppLauncher 模式是刚需**: IsaacSim 要求在导入任何 IsaacLab 模块前先启动 Omniverse，忘了这一步会导致 `omni.client` 等导入错误。所有脚本必须遵循「解析参数 → AppLauncher → 其他 import」的严格顺序。

2. **Gym 注册用字符串引用**: IsaacLab 的 gym 注册 `env_cfg_entry_point` 不能直接用类对象（会触发 IsaacSim 提前导入），必须用 `f"{__name__}.module:ClassName"` 字符串延迟加载。

3. **PBRS 的状态跟踪**: Potential-Based Reward Shaping 需要记住上一步的势函数值 Φ(s)。在 IsaacLab 的 `RewTerm` 函数式 API 下，只能通过给 `env` 动态添加属性 (`env._pbrs_prev_potential`) 来实现。需要特别注意 episode reset 时清零。

4. **`reward_manager._step_reward`**: IsaacLab 的 reward manager 在每一步都会填充 `_step_reward` (shape: `num_envs × num_terms`)，是获取逐项奖励值的最可靠方式。`extras["log"]` 只在 episode reset 时才有值。

5. **随机动作下奖励幅值小是正常的**: shaped 奖励设计要求 agent 依次完成 reach → grasp → lift → hold，随机动作很难触发后续阶段，所以 `check_reward_range.py` 报告的幅值很小 (|max|≈0.0007) 是符合预期的。正式训练中随着策略改善，奖励会显著增大。

6. **显存约束的实际效果**: 严格遵守 `--headless` + `num_envs ≤ 4` (冒烟) / `≤ 128` (训练) 的规则，全天未触发 OOM。RTX 4060 Laptop 8GB 完全可以支撑 4 env 的快速验证循环。

7. **一天四个 Phase 是可行的**: 有完善的文档体系 (process.md + verify_checklist.md + checkpoint.md) 支撑，AI 可以高效地自主驱动 "实现 → 验证 → 记录 → 推进" 的闭环，关键是每个 Phase 都有明确的验收标准。

8. **force push 要小心**: 将本地分支 force push 到远程仓库时，会覆盖原有文件（如 README.md）。更安全的做法是先 `git fetch` + `git merge` 再推送，或至少提前备份远程独有的文件。
